## Assignment

Research knowledge management approaches for issue #321. Investigate how to manage the growing `.squad/` knowledge base without binary bloat or losing GitHub-native workflows.

## Work Summary

### Research Scope
Investigated 6 domains:
1. Current state: 33MB total (.squad/), 29.5MB compiled .NET binaries, ~3.5MB markdown
2. AI agent memory patterns (LangChain, AutoGPT, CrewAI)
3. GitHub-native options (Wiki, Pages, Discussions, Search)
4. Lightweight databases (SQLite, LMDB, ChromaDB, FAISS)
5. Git storage strategies (git-annex, git-LFS alternatives)
6. Community practices from developer forums

### Key Findings
- **Vector databases** (ChromaDB, Qdrant, FAISS) are standard for semantic search
- **Hybrid storage**: Vector DB for search + structured DB for metadata
- **Git-friendly approach**: Store schema/exports, not binaries; avoid binary DB files in git
- **Best practices**: Versioned markdown directories + search indexing (LangChain pattern)

### Deliverables
- Comprehensive research report posted to GitHub issue #321
- 5 approaches ranked by feasibility:
  1. Structured markdown + archival (Phase 1, immediate, zero new tooling)
  2. Vector DB indexing (Phase 2, future, if semantic search becomes critical)
  3. git-annex for large historical data
  4. GitHub Pages wiki with search
  5. Hybrid SQLite + scheduled exports

### Recommendation
**Two-phase approach:**
- **Phase 1:** Rotate agent history files quarterly; archive resolved decisions by quarter; exclude build artifacts (saves 29.5MB immediately)
- **Phase 2:** Add ChromaDB indexing when markdown corpus exceeds ~50MB
