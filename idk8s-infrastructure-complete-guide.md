## 1. Executive Summary

The **idk8s-infrastructure** repository is the operational backbone of **Project Celestial** — Microsoft Entra (Identity) division's Kubernetes-native fleet management platform (v3). Owned by the Identity Platform Team within Microsoft's Identity division (Service Tree ID: `5cee095b-4378-4308-b09d-06b70ae9ff8a`), this monorepo houses the Management Plane, Component Deployer, Automated OS (AOS) orchestration, FleetManager SDK, and the EV2-based release pipeline infrastructure for deploying identity workloads (DPX, Gateway, and 19 tenant services) across AKS clusters spanning Azure Public, Fairfax (Gov), Mooncake (China), USNat, USSec, BLEU, and Delos sovereign clouds. It is the successor to legacy IAMK8s (v2) and the LSM-based deployment model, converging toward a fully Kubernetes-native, operator-driven control plane.

## 2. Motivation & Vision

### 2.1 Why Does This Platform Exist?

Microsoft's Identity division (Entra) runs critical authentication and authorization services that must be globally distributed, highly available, and deployed safely across dozens of Azure regions and multiple sovereign clouds. Before Celestial, these workloads were managed through **LSM (Legacy Service Management)** — a deployment system that predates Kubernetes and lacked the declarative, self-healing properties modern infrastructure demands.

The core problems Celestial solves:

- **Fleet-scale Kubernetes management**: Operating dozens of AKS clusters across 7+ cloud environments with consistent configuration, health monitoring, and deployment practices
- **Safe multi-tenant workload deployment**: 19 distinct tenant teams deploy their workloads onto shared cluster infrastructure, each requiring isolation, identity management, and independent release cadences
- **Unified deployment pipeline**: A single EV2-integrated deployment system replacing fragmented LSM-based processes, with built-in safe deployment practices (rings, canary, rollback)
- **Automated node and pod health management**: Proactive detection and remediation of infrastructure issues before they impact identity services

### 2.2 Strategic Direction

From the Management Plane tech debt and vision document (`/docs/management-plane/MP-tech-debt-and-vision.md`), the platform's strategic roadmap includes:

1. **SU deployment via MP (LSM substitution)** — The Scale Unit API replaces LSM as the primary fleet-wide deployment mechanism. Tenants migrate from `DeploymentWorkflowVersion: "1.0"` (LSM) to `"2.0"` (SU API).
2. **Entra-free SU deployments** — Transitioning from Entra ID (AAD) to dSTS (decentralized STS) for authentication, removing dependency on Entra infrastructure for deploying Entra services (circular dependency elimination).
3. **CRD-based state management (MVP Phase 3)** — Migrating from ConfigMap-based persistent storage to Kubernetes Custom Resource Definitions, enabling structured, observable, and API-native state management.
4. **Cluster RP Operator** — A Kubernetes operator wrapping `idk8sctl` for on-demand cluster provisioning, eliminating manual cluster creation workflows.
5. **N+1 Cluster Upgrade Orchestration** — Automated disruptive cluster upgrades using a spare-cluster strategy to maintain availability during major version transitions.
6. **Workload Migration Controller** — Automated workload migration between clusters for maintenance, rebalancing, or decommissioning scenarios.
7. **Management Plane SLOs** — Formal service level objectives for MP availability and deployment latency.
8. **Per-component versioning** — Moving from monolithic `idk8sctl` version tracking to individual component digests for granular deployment control.

### 2.3 Evolution from LSM to Kubernetes-Native Fleet Management

The platform evolution follows three major phases:

| Phase | Era | Model |
|-------|-----|-------|
| **v1** | IAMK8s | Manual/script-based cluster operations |
| **v2** | IAMK8s + LSM | LSM-managed deployments with shared Nginx ingress + Linkerd service mesh |
| **v3** | **Celestial** (current) | Kubernetes-native fleet management with dedicated IPs per partner, EV2-integrated deployment, operator-pattern reconciliation |

Key architectural shifts in v3:
- **Shared infrastructure → Dedicated resources**: Each partner gets dedicated IP addresses, certificates, and namespaces (no shared Nginx/Linkerd)
- **Imperative → Declarative**: Spec/Status/Generation pattern for all entities, with reconciliation loops converging desired → actual state
- **LSM → SU API**: EV2 HTTP extensions replace LSM deployment jobs
- **Entra → dSTS**: Certificate-based authentication removing circular dependency on Entra
- **ConfigMaps → CRDs** (planned): Structured, typed state management replacing opaque ConfigMap storage

## 3. Use Cases & Functionality

### 3.1 What Can You Do with This Platform?

The idk8s-infrastructure platform provides the following capabilities:

#### Deploy a Scale Unit
A scale unit is an isolated tenant workload instance deployed to a specific cluster in a specific region. The deployment flow:
1. Tenant configures their service in `ServiceProfile.json` under `ResourceProvider/Data/Tenants/`
2. Tenant's build pipeline generates EV2 spec with `DeploymentWorkflowVersion: "2.0"`
3. EV2 calls the Management Plane's SU API HTTP extension endpoints
4. The ScaleUnit Reconciler schedules the workload onto an appropriate cluster
5. Helm charts are deployed, namespaces created, identities provisioned

#### Create and Manage Clusters
Using `idk8sctl` (from the `idk8s-infrastructure-enablement` repo) and the Component Deployer:
1. Provision AKS clusters via Bicep templates (`src/bicep/`)
2. Deploy cluster-level components (infrastructure, platform) via Component Deployer manifests (`src/manifests/`)
3. Monitor cluster health via AOS components (NHA, NPD, NRS)

#### Onboard a New Tenant
Following the process in `/docs/management-plane/SU-API-Onboarding.md`:
1. **Celestial team**: Creates Entra app registration per environment, adds deployment identity mapping in `ScaleUnits.json`, authorizes against SU API app registrations
2. **Customer team**: Updates build pipeline to use `DeploymentWorkflowVersion: "2.0"`, adds `InlineDynamicBindingOverrides` with application ID to release pipeline
3. Tenant data (service profiles, scale unit definitions) added under `ResourceProvider/Data/Tenants/{tenant}/`

#### Manage Cluster Components
The Component Deployer processes YAML manifests (`src/manifests/`) that declare cluster-level components with:
- Dependency DAGs (components execute in dependency order)
- Topology selectors (deploy components conditionally by ring, environment, region)
- Three cluster types: DPX, Gateway, Generic — each with its own manifest

#### Monitor and Remediate Node Health
AOS components provide automated health pipeline:
- **Node Health Agent**: Bridges Azure VMSS scheduled events → Kubernetes node conditions
- **Node Problem Detector**: Detects hardware/OS-level problems
- **Node Repair Service**: Taints/untaints nodes based on health conditions
- **Pod Health Check Service + API Controller**: Pod-level health monitoring and API exposure

### 3.3 Key Workflows

#### Deploy a Scale Unit (EV2 Flow)
```
ADO Build Pipeline
  → Docker build (images → ACR)
  → GenerateReleaseSpec (DeploymentWorkflowVersion: "2.0")
  → Publish artifacts

ADO Release Pipeline
  → ExpressV2Internal task
  → InlineDynamicBindingOverrides (Application ID)
  → EV2 Safe Deployment (rings, canary)

EV2 → Management Plane
  → POST /deploy (initiate deployment)
  → GET /status (poll progress)

Management Plane → ResourceProvider
  → Authenticate (S2S cert, Entra or dSTS)
  → Authorize (tenant DeploymentIdentities check)
  → Create/update ScaleUnit entity in Inventory
  → Trigger ReconciliationRunner

Reconciliation
  → ScaleUnitReconciler: schedule, deploy Helm release
  → DeploymentUnitReconciler: provision namespace, identities, Azure resources, Helm install
```

#### Create a Cluster
```
1. Model subscription in ad-FUN-idsr (AC2)
2. Create regional resources (Key Vault, certs, Entra apps) in IDK8s-MP.ac2
3. Apply via LSM job in Geneva Actions
4. Create OS images in idk8s-infrastructure-enablement
5. Add EV2 spec + cluster config in idk8s-infrastructure
6. Trigger MP cluster creation pipeline
7. Deploy SU API (13-step process)
```

#### Onboard a Tenant
```
1. Create App Registration in Azure portal (per environment)
2. Configure manifest with trustedCertificateSubjects
3. Authorize app against SU API app registrations
4. Add DeploymentIdentities mapping in ScaleUnits.json
5. Provide Application ID to customer
6. Customer updates build pipeline (DeploymentWorkflowVersion: "2.0")
7. Customer updates release pipeline (InlineDynamicBindingOverrides)
```

### 3.4 Runtime Modes

The Management Plane / Scale Unit API operates in two modes, controlled by the `ManagementPlane__Mode` environment variable:

| Mode | Purpose | Status |
|------|---------|--------|
| **Generic (EV2) mode** | Production mode. Exposes EV2-compatible HTTP extension endpoints for scale unit deployment and monitoring. EV2 calls these endpoints during safe deployment rollouts. | ✅ Active |
| **CLI mode** | Development sandbox mode. Backend for Celestial CLI operations on developer scale units. | ⚠️ Currently not in active use |

In **Generic mode**, the SU API exposes:
- `POST /deploy` — Initiate scale unit deployment
- `GET /status` — Poll deployment progress
- `/validate`, `/suspend`, `/cancel` — Not yet implemented (noted as tech debt)

### 4.2 Regional Deployment Model

Each Azure region hosts a dedicated **Management Plane cluster** that serves as the control plane for all workload clusters in that region. This is a one-MP-per-region model:

```
┌──── Azure Region: Central US ────────────────────────────────┐
│                                                               │
│   ┌─────────────────────┐                                     │
│   │  MP Cluster (CUS)   │ ◄── EV2 HTTP Extension calls       │
│   │  - Scale Unit API   │                                     │
│   │  - FleetManager SDK │                                     │
│   └────────┬────────────┘                                     │
│            │ manages                                          │
│            ▼                                                  │
│   ┌──────────────┐  ┌──────────────┐  ┌──────────────┐       │
│   │ DPX Cluster  │  │ Gateway      │  │ Generic      │       │
│   │ (workloads)  │  │ Cluster      │  │ Cluster      │       │
│   └──────────────┘  └──────────────┘  └──────────────┘       │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

MP clusters are integrated into the main release pipeline alongside workload clusters (ADR-0005), deployed in parallel during Ring 1 and Ring 2 rollouts. This eliminates synchronization issues and configuration drift that occurred with independent deployment processes.

### 4.3 Data Flow: EV2 → MP → Cluster → Components

#### Scale Unit (Workload) Deployment

```
Step 1: BUILD
  ADO Pipeline → Docker images → ACR
               → EV2 spec (DeploymentWorkflowVersion: "2.0")
               → Publish artifacts

Step 2: RELEASE
  ADO Release → ExpressV2Internal task
             → InlineDynamicBindingOverrides (App ID for auth)
             → EV2 Safe Deployment Practices (rings, canary)

Step 3: EV2 → MANAGEMENT PLANE
  EV2 HTTP Extension → POST /deploy (initiate)
                     → GET /status (poll)
  Platform-generated cert → Entra token → MISE validates

Step 4: MANAGEMENT PLANE → RESOURCE PROVIDER
  Authenticate (S2S cert via Entra or dSTS)
  → Authorize (tenant DeploymentIdentities check)
  → Create/update ScaleUnit entity in Inventory
  → ReconciliationRunner triggers

Step 5: RECONCILIATION
  ScaleUnitReconciler:
    → Check Generation != ObservedGeneration
    → AddScaleUnit → DeployRelease (Helm)
    → Update ObservedGeneration

  ScaleUnitScheduler:
    → Filter clusters (topology, mappings)
    → Score clusters (spread scoring)
    → Create DeploymentUnit on best cluster

  DeploymentUnitReconciler:
    → Create identity group, namespace, quotas
    → Provision Azure resources (KeyVault, Storage)
    → Deploy Helm release to target cluster
    → Update status (AppliedSpecHash, DeployedReleaseId)

Step 6: KUBERNETES CLUSTER
  Target cluster receives:
    → New namespace for the scale unit
    → Resource quotas applied
    → Managed identity configured
    → Helm chart installed/upgraded
    → Pods scheduled and running
```

#### Cluster Infrastructure Deployment

```
EV2 (per-cluster stamp, ADR-0006)
  → Shell Extension invokes ComponentDeployer.CLI
  → Reads cluster-type manifest (dpx/gateway/generic)
  → Resolves component dependency DAG
  → Executes plugins per component kind:
      - Idk8sctlBicep (ARM/Bicep deployment)
      - Idk8sctlConfigure (default cluster components)
      - Idk8sctlConfigureComponent (single component)
  → Topology selector filters by ring/env/region
  → Tracks release state in Azure Storage Tables
  → Reports status back to EV2
```


The multi-cloud strategy drives the dSTS migration: sovereign clouds cannot use Entra ID, making dSTS the universal authentication mechanism across all environments.

### 4.5 Kubernetes-Native Resource Model

The system follows a **Kubernetes-native resource model** even outside Kubernetes itself. All managed entities use the canonical Spec/Status/Generation pattern:

```
┌──────────────────────────────────────────────────┐
│  Entity (ScaleUnit, DeploymentUnit, etc.)        │
│                                                   │
│  Metadata:                                        │
│    ├── Generation          (incremented on spec   │
│    │                        change)               │
│    └── ResourceVersion     (optimistic concurrency│
│                             via etcd semantics)   │
│                                                   │
│  Spec:        (desired state — what the user      │
│                wants)                             │
│                                                   │
│  Status:                                          │
│    ├── ObservedGeneration  (last reconciled       │
│    │                        generation)           │
│    ├── AppliedSpecHash     (XxHash32 of applied   │
│    │                        spec, for DU)         │
│    └── DeployedReleaseId   (current Helm release) │
└──────────────────────────────────────────────────┘
```

**Reconciliation pattern:**
- If `Metadata.Generation == Status.ObservedGeneration` → no work needed (idempotent)
- If they differ → reconciler processes the entity, converging actual → desired state
- `ResourceVersion` provides optimistic concurrency (throws `ResourceVersionConflictException` on stale writes)
- Reconcilers can create/update entities that generate cascading events for subsequent reconciliation rounds

**Inventory system** (`IInventory<T>`) acts as the "etcd" of this system:
- Namespace-scoped entities (`InventoryNamespace` + `InventoryObjectName`)
- CRUD + query with optimistic concurrency
- Implementations: `InMemoryInventory` (dev/test), `AzureTableStorage` (production)
- Event-driven: `IInventoryEventSink` notifies `ReconciliationRunner` on entity changes

**Scheduler** (Filter → Score → Select):
- `TopologyClusterFilter`: Match cloud, environment, region, cluster type
- `MappingBasedFilter`: Honor explicit SU→cluster mappings from `ScaleUnitClusterMappingsConfiguration.json`
- `SpreadScaleUnitsScorer`: Prefer clusters with fewer scale units
- Supports explicit pinning via pattern-based and exact-name mappings

## 7. Infrastructure & Deployment

### 7.1 Helm Chart Architecture

#### Celestial Chart Structure

The primary workload chart lives at `/src/helm/charts/celestial/` and serves as a **platform-level shared template** rendered per-service by Fleet Manager at deployment time. It is not a per-service chart — every Celestial workload deploys through this single chart with different values.

**Chart metadata** (`Chart.yaml`): API version v2, chart version 1.0.0, type Application.

##### Template Inventory

| Template | Purpose |
|----------|---------|
| `_helpers.tpl` | Name generators, labels, security context, cloud-to-Geneva mapping, cluster version detection |
| `deployment.yaml` | Core template (~400 lines) — multi-container Deployment with Geneva sidecars, dSMS support, init containers |
| `service.yaml` | LoadBalancer Service (only for `external` type with service tag annotations) |
| `hpa.yaml` | Standard HPA (autoscaling/v2) for CPU/memory targets |
| `keda.yaml` | KEDA ScaledObject for advanced autoscaling (v3 clusters only) |
| `pdb.yaml` | PodDisruptionBudget with `AlwaysAllow` unhealthy pod eviction |
| `_autoscale_policies.yaml` | KEDA HPA behavior policies (scaleUp/scaleDown) |
| `_autoscale_profile.yaml` | KEDA trigger profiles (cron, CPU, memory) |

##### Deployment Template Architecture

Each pod has 3–5 containers with a specific lifecycle coordination pattern:

1. **Init Container: `init-geneva-cert`** (v2/platform-managed secrets) — Uses Azure CLI + workload identity to download Geneva and SSL certificates from KeyVault, converts PFX → PEM
2. **Init Container: `dsmsnativebootstrapper`** (v3/dSMS mode) — Runs dSMS native bootstrapper with `restartPolicy: Always` for continuous certificate rotation
3. **Sidecar: `mdsd`** (Geneva Logs) — Configurable size (small/medium/large/xLarge/2xLarge) with graceful shutdown coordination
4. **Sidecar: `mdm`** (Geneva Metrics) — Metrics agent with socket-based IFX input
5. **Application container** — Single app container (multi-container explicitly blocked via fail guard)

**Graceful shutdown coordination**: The app container creates `MainAppContainerDied.txt` in a shared volume on termination. Geneva sidecars poll for this file before starting their flush timers. `terminationGracePeriodSeconds` = eventsFlushDuration + maxAppTerminationGrace + 60s buffer.

### 7.3 Component Deployer System

#### Manifest Schema and Kinds

**Location**: `/src/manifests/`

All component manifests share a common YAML schema with a `kind` field and a dependency chain:

| Kind | Description |
|------|-------------|
| `idk8sctlbicep` | Bicep infrastructure deployment via `idk8sctl create` |

#### 3 Manifest Types

| Manifest | Cluster Kind | Components |
|----------|-------------|------------|
| `generic-manifest.yml` | `generic` | Base set: 7 components |
| `gateway-manifest.yml` | `gateway` | Base + wireserver-file-proxy: 8 components |
| `dpx-manifest.yml` | `dpx` | Base + skylarc-agent: 8 components |

#### Dependency DAG and Execution Plan

Components are deployed in stages based on their dependency chain. Parallel execution occurs within each stage; stages execute sequentially:

```
Stage 1: cluster-bicep-model (idk8sctlbicep)
    │
Stage 2: core-cluster-components (idk8sctlconfigure)
    │
Stage 3 (parallel):
    ├── node-health-agent (IS)
    ├── node-repair-service (IS)
    ├── descheduler (IS)
    ├── node-problem-detector (IS)
    ├── keda (WS)
    ├── pod-health-check-service (WS)
    ├── remediation-controller (WS)
    ├── wireserver-file-proxy (WS) [gateway only]
    └── skylarc-agent (WE) [dpx only]
```

**Team ownership**: IE owns foundation (bicep, core-cluster), IS owns node services (NHA, NRS, descheduler, NPD), WS owns workload services (PHCS, RC, KEDA, WSP), WE owns workload engineering (Skylarc).

#### State Management

Component Deployer tracks cluster state to optimize deployments:

- During installation, stores **cluster state** including versions of all installed components in **Azure Storage Account Tables**
- Before deploying, compares **desired version** with **actual version** — skips if they match
- Storage accounts per environment: `inventorysadev` (Dev), `inventorysatest` (Test), `inventorysaprod` (Prod)
- Current implementation uses the **idk8sctl artifact version** as the component version for all components
- Future: Each component will have its own unique version (e.g., image digest) for granular skip logic

#### Topology Selector

The topology selector enables conditional component installation based on deployment context. The filter is applied **before** building the execution plan, with dependency validation after filtering.

**Supported keys**: `ring` (e.g., prod-0, prod-1), `environment` (prod, test), `region` (eastus, centralus)

**Supported operators**: `In` (value must be one of specified values), `NotIn` (value must not be any of specified values)

**Match expressions** use AND logic — all conditions must be satisfied:

```yaml
topologySelector:
  matchExpressions:
    - key: ring
      operator: In
      values: ["prod-0"]
    - key: environment
      operator: In
      values: ["prod"]
```

> ⚠️ If a topology selector filters out a component that other components depend on, the deployment fails with an exception. Dependencies are validated after filtering.

## 8. Security Architecture

### 8.1 Authentication Flows

#### MISE Middleware (Microsoft Identity Service Essentials)

Production authentication uses MISE middleware with two configurable modes:

| Mode | Config Section | Trigger Condition |
|------|---------------|-------------------|
| **dSTS path** | `"dSTS"` | `managementPlaneConfiguration.UseDsts == true` |
| **Entra path** | `"AzureAd"` / `"AzureAdCli"` | Default; CLI mode uses `AzureAdCli` section |

Both modes register ASP.NET Core authentication schemes via `AddMiseWithDefaultAuthentication` or `AddMiseWithDefaultModules`. The dSTS path additionally enables `TokenAcquisitionToCallDownstreamApi` for S2S authentication.

**Source**: `/src/csharp/fleet-manager/ManagementPlane/Program.cs`

#### EV2 Certificate Exchange Flow

EV2 authenticates to the SU API through a certificate exchange:

```
EV2 Platform → Downloads cert (compound identity from onboarding)
    → Exchanges cert for Entra token (audience: https://idk8s-managementplane-{env})
        → SU API validates token via MISE
            → TenantAuthorizationHandler verifies OID matches ServiceProfile.json
```

The EV2 extension manifest specifies `AadApplicationAuthentication` with a per-environment resource URI.

**Source**: `/src/ev2/extensions/MPExtensionManifest-PROD.json`

#### Entra ID Application Per Cloud/Env

Each cloud/environment combination has a dedicated Entra ID application registration. The Management Plane authenticates to downstream services using a **OneCert SN+I certificate** stored in Azure Key Vault, accessed via Node MSI and used to create `ClientCertificateCredential`-backed `ArmClient` instances.

**Source**: `/src/csharp/fleet-manager/ResourceProvider/AzureClient/AzureClientProvider.cs`

#### dSTS (Future State)

dSTS (Distributed Security Token Service) provides node-level authentication via Service Client Identities (SCIs). The DsmsBootstrapNodeService authenticates to dSMS using MSAL with `sendX5C: true` for certificate chain validation. SCI info is resolved from `foundational_resources.json` (new path) or legacy `dmsi.ini` + `SciData.json`. The constant dSTS tenant ID is `7a433bfc-2514-4697-b467-e0933190487f`.

**Source**: `/src/csharp/fleet-manager/DsmsBootstrapNodeService/Services/DsmsClient.cs`

#### Mock Authentication for Development

A `MockAuthenticationHandler` is enabled only when `isDevelopment` is true. It creates synthetic claims (UniqueNameClaimName, IdTypClaimName as `"app"`, OidClaimName). All endpoints still require authorization — the mock handler merely bypasses the token validation.

**Source**: `/src/csharp/fleet-manager/ManagementPlane/Auth/MockAuthenticationHandler.cs`

### 8.2 Authorization Model

#### TenantAuthorizationHandler

Two authorization handlers map caller identity to Celestial tenants:

### 8.3 Certificate Management

#### Certificate Lifecycle Overview

```
OneCert (SN+I) → Azure Key Vault (per-region) → Node MSI (manual) → MP Program (startup)
dSMS Cert (per-VMSS) → VMSS ARM Template → KeyGuard (validated) → DsmsBootstrapNodeService
ACMS Agent → DsmsNativeBootstrapper (init container) → OneCert Rotation (automated)
SSL/Geneva Certs → AKV (platform-managed) OR dSMS (v3) → init-geneva-cert container
EV2 Cert (platform) → Compound Identity → Entra Token Exchange
```

#### DSMSBootstrapNodeService

**DaemonSet-style service** running on each k8s worker node:

1. Monitors containerd for new containers with `dsms/containers` annotations
2. Downloads certificates from dSMS using dSTS tokens
3. Injects certificates into containers via `/dev/shm/dsms/` (tmpfs memory-backed)
4. Certificate and password written as separate files: `bootstrapcert.pfx` and `bootstrapcert.pass`

**Source**: `/src/csharp/fleet-manager/DsmsBootstrapNodeService/`

#### KeyVaultWatcher

Monitors Azure Key Vault for secret changes. Configured via ConfigMap with JSON specifying:
- `KeyVaultsToWatch` — list of KV URLs and secrets to monitor
- `SecretsToWatch` — per-secret type (Certificate/secret) with target deployments to restart
- `IntervalSeconds: 300` — 5-minute polling interval

On secret rotation, triggers rolling restart of specified Deployment resources.

#### SN+I (Subject Name + Issuer) Certificate Auth

The Management Plane uses Subject Name + Issuer certificates for downstream service authentication. The certificate is loaded from AKV at startup via `CertificateLoader.LoadCertificate()` and used to create `ClientCertificateCredential` instances for ARM API calls.

#### KeyGuard Validation

The `KeyGuardValidationService` proactively validates that certificate private keys are stored in KeyGuard (hardware-backed VBS isolation). Checks the `NCryptUseVirtualIsolationProperty` on RSACng keys. Validates two certificate types: AgentID (self-signed) and VmssID (cluster cert). Emits metrics for monitoring.

**Source**: `/src/csharp/fleet-manager/DsmsBootstrapNodeService/Services/KeyGuardValidationService.cs`

#### In-Cluster TLS Management (ADR-0007)

ADR-0007 addresses TLS certificate management within clusters. OSS components (Gatekeeper, metrics-server, KEDA) use built-in self-signed CA bundles — acknowledged as a known gap with security team follow-up pending. Platform components use OneCert-issued certificates managed through the DSMS/ACMS pipeline.

### 8.4 Network Security

#### Load Balancer Source Restrictions

SU API endpoints allow connections **only from EV2 IPs** as configured in values.yaml files. The service template uses `loadBalancerSourceRanges` and `azure-pip-ip-tags` annotations for FirstPartyUsage tagging.

#### NSG for MP Cluster Access

Management Plane clusters are network-restricted to EV2 source IPs. The `service.yaml` template supports `loadBalancerSourceRanges` via values files.

#### No SAW Access (Known Limitation)

EV2 does not support service tags, and the Azure Cloud plugin cannot mix service tags with individual IPs in `loadBalancerSourceRanges`. This means:
- **No SAW access** for operational debugging of production SU API
- No additional allowed service tags for monitoring/management
- Recommendation: Investigate Azure Private Link as an alternative access path

#### WireServer Access

WireServer (169.254.169.1) is blocked at the network level. A dedicated `WireServerFileProxy` (ADR-0011) runs as root to proxy only the service tags file. IMDS access is prohibited by default with an explicit exception only for the ScheduledEventsCollector (ADR-0002).

### 8.5 Container Security

#### Hardened Security Context

Applied to **all** containers via the `celestial.containerSecurityContext` helper:

```yaml
securityContext:
  allowPrivilegeEscalation: false
  capabilities:
    drop: [all]
  runAsNonRoot: true
  runAsUser: 1000
  runAsGroup: 1000
  readOnlyRootFilesystem: true
  seccompProfile:
    type: RuntimeDefault
```

**Service Account Token**: `automountServiceAccountToken: false` for service deployments; `true` only for KeyVaultWatcher (requires k8s API access).

#### Memory-Backed Secret Volumes

All sensitive volumes use memory-backed emptyDir (`medium: "Memory"`) with explicit size limits to prevent secrets from touching disk:
- `keyvault-auth-vol`: 10Mi (Geneva certs)
- `ssl-auth-vol`: 10Mi (SSL certs)
- `dsms-secrets`: 5Mi (dSMS secrets, v3 only)

#### KeyGuard Validation

Proactive monitoring via `KeyGuardValidationService` ensures certificate private keys are stored in hardware-backed VBS (Virtualization-Based Security) isolation. Validates both AgentID and VmssID certificates with metrics emission for alerting.

## 10. Architecture Decision Records (ADRs)

All ADRs are stored in `/docs/adr/` and follow Michael Nygard's lightweight format (adopted per ADR-0001). ADRs are accepted via PR review; superseded ADRs reference their successors.

### ADR-0001: Record Architecture Decisions

| Field | Value |
|-------|-------|
| **Date** | April 19, 2024 |
| **Status** | Accepted |
| **Context** | The team needed a lightweight mechanism to capture significant architectural choices, their rationale, and consequences without requiring heavyweight design documents for every decision. |
| **Decision** | Adopt Architecture Decision Records following Michael Nygard's format. ADRs are stored in `docs/adr/`, accepted via PR review. Superseded ADRs receive new ADRs that reference the old ones. Detailed design docs may still be requested for complex decisions. |
| **Consequences** | Requires team discipline to author ADRs for significant decisions. Supplements (does not replace) detailed technical specifications. Provides a searchable, version-controlled history of architectural choices. |
| **Related Components** | All — meta-process for the entire repository |

### ADR-0002: IMDS Exception for Scheduled Events Collector (SEC)

| Field | Value |
|-------|-------|
| **Date** | March 5, 2025 |
| **Status** | Accepted |
| **Context** | VMs in VMSS receive scheduled maintenance events via the Instance Metadata Service (IMDS). The ScheduledEventsCollector (SEC) component needs to listen to IMDS to detect upcoming events and persist them as Kubernetes node conditions, enabling proactive workload draining before maintenance. However, IMDS endpoint access is prohibited by default for security reasons. |
| **Decision** | Develop SEC to listen to the IMDS endpoint for upcoming scheduled events and persist them as Kubernetes node conditions. A security exception must be filed to grant SEC access to IMDS. |
| **Consequences** | Requires a formal security exception for IMDS access. The exception is properly scoped to SEC only — default-deny is maintained for all other workloads. |
| **Related Components** | `NodeHealthAgent`, `ScheduledEventsCollector`, `IMDS`/`IMDSProxy` (`/src/csharp/fleet-manager/IMDS/`) |

### ADR-0003: Platform-Managed Certificates for Partners

| Field | Value |
|-------|-------|
| **Date** | February 25, 2025 |
| **Status** | Accepted |
| **Context** | In v2, shared Nginx ingress + Linkerd handled TLS and a shared Geneva account handled logging. In v3 (Celestial), each partner gets a dedicated IP with no shared infrastructure, raising the question of who manages SSL and Geneva certificates. |
| **Decision** | The Celestial platform manages SSL and Geneva certificates for partners (maintaining the v2 contract) rather than requiring BYO certificates. Certificates are generated and stored in platform-owned Key Vaults. |
| **Consequences** | Platform must ensure secure certificate generation, storage, and delivery. Auto-rotation must work without partner intervention. Security team review may result in potential reversal. |
| **Related Components** | `DsmsBootstrapNodeService`, `DsmsNativeBootstrapper`, Key Vault Watcher (`/src/helm/charts/keyvaultwatcher/`), init-geneva-cert container |

### ADR-0004: Deployment Process for Cluster Release Candidates

| Field | Value |
|-------|-------|
| **Date** | May 29, 2025 |
| **Status** | **Superseded** by ADR-0007b |
| **Context** | The team needed a process for the InfraEnablement (IE) team to nominate commits and publish cluster release artifacts for consumption by the release pipeline. |
| **Decision** | Build ID-based flow where IE nominates commits, publishes artifacts via pipeline, and the release pipeline consumes them by pinned build number. Artifacts include Bicep templates, components, and executables published to `drop_publish_idk8sctl_bundle/`. |

### ADR-0005: Management Plane Clusters in Release Pipeline

| Field | Value |
|-------|-------|
| **Date** | June 3, 2025 |
| **Status** | Accepted |
| **Context** | MP clusters are regionally scoped (one per region), hosting the SU API. They were deployed independently from workload clusters, causing synchronization issues and operational drift. |

### ADR-0006: Cluster Orchestrator — Cluster vs. Regional Mode

| Field | Value |
|-------|-------|
| **Date** | June 5, 2025 |
| **Status** | Accepted |
| **Context** | The team needed to decide whether EV2 should treat deployments at the region level (Region Orchestrator) or per-cluster level (Cluster Orchestrator). |
| **Decision** | Adopt the **Cluster Orchestrator** model where every cluster is a separate EV2 stamp. This offloads parallelism, retries, LKG tracking, and visibility to EV2. |

### ADR-0007a: In-Cluster TLS Certificate Management

| Field | Value |
|-------|-------|
| **Date** | June 27, 2025 |
| **Status** | Accepted |
| **Context** | Custom cluster components need TLS certificates for secure in-cluster communication. Options included distinct namespaces, different cluster-local domains, DNS rewrite rules, and cluster CA with cert-manager. |
| **Decision** | Use **distinct namespaces per component** with OneCert for AME certificates (e.g., `skylarc-dpx-prod-celestial`) as a short-to-medium-term solution. |
| **Consequences** | Services must parameterize target DNS names. OSS components (Gatekeeper, metrics-server, KEDA) using self-signed CAs remain an acknowledged gap. Follow-up needed with the security team on Cluster CA usage. |

### ADR-0007b: Tag-Based Deployment Process for Release Candidates

| Field | Value |
|-------|-------|
| **Date** | June 12, 2025 |
| **Status** | Accepted (supersedes ADR-0004) |
| **Context** | ADR-0004's build-ID approach required a PR for every release to update the build number — excessive overhead. |
| **Decision** | Adopt semantic versioning (`cluster/vX.Y` tags) for cluster release artifacts. The release pipeline discovers artifacts by tag rather than build ID. Major = breaking changes (cluster rebuilds); Minor = compatible features; Patch = compatible fixes. |
| **Consequences** | IE team tags → release pipeline pulls latest successful run with matching tag. Cannot use different artifacts for different cluster types (single tag) — to be addressed later. |
| **Related Components** | `idk8s-infrastructure-enablement` pipeline (436735), cluster release pipeline (414565) |

### ADR-0008: Release Branch Process for Cluster Configurations

| Field | Value |
|-------|-------|
| **Date** | July 1, 2025 |
| **Status** | Accepted |
| **Context** | When a cluster release needs to be restarted after config changes have landed on `main`, using `main` for the restart is error-prone (picks up unintended changes). |
| **Decision** | Create hotfix branches **on demand** (`hotfix/vX.Y`) from the original commit, not proactively per release cycle. Pipeline runs from the hotfix branch; changes are cherry-picked back to main. |

## 12. Operational Model

### 12.1 AOS (Autonomous Operating System) Overview

The AOS subsystem manages automated node health monitoring, scheduled event coordination, and remediation for all Celestial clusters. It is a three-component pipeline ensuring VMs stay healthy and maintenance events are handled gracefully.

**Source:** `/docs/aos/aos-failure-modes.md` (created September 2, 2025)

**Pipeline flow:**

```
VMSS Scheduled Events → SEC/NHA → K8s Node Conditions → NRS → Taints → Pod Draining
                                                          ↑
Azure Node Health    → NPD ─────→ K8s Node Conditions ───┘
```

**Component responsibilities:**

| Component | Role | Deployment |
|-----------|------|------------|
| **Scheduled Events Collector (SEC)** | Listens to IMDS for upcoming scheduled maintenance events | Part of NHA pod |
| **Node Health Agent (NHA)** | Channels VMSS events to K8s API; reports health status back to VMSS Resource Provider | DaemonSet (per-node, Linux + Windows images) |
| **Node Problem Detector (NPD)** | Detects and reports node problems as K8s node conditions | DaemonSet (per-node, Windows image from external source) |
| **Node Repair Service (NRS)** | Taints/untaints nodes based on NHA + NPD health conditions | Deployment (cluster-level replicas) |

### 12.2 Failure Modes (5 Documented Scenarios)

#### Failure Mode 1: Node Health Agent Down on a Node

- **Effects:** Scheduled event conditions not reported; NRS cannot taint for maintenance; VMSS RP AHE reports "Unknown"
- **Causes:** NHA process crash, misconfiguration, deadlock, image pull failure, pod eviction
- **Detection gaps:** Generic monitoring is too noisy; no specific alert for NHA absence; no cross-check with VMSS scheduled events
- **Action items:** Set `system-node-critical` priority class; configure adequate resource requests/limits; add SEC initialization wait in readiness probes; use Kubernetes Graceful Node Shutdown

#### Failure Mode 2: Node Problem Detector Down on a Node

- **Effects:** Health conditions not reported; unhealthy nodes not tainted
- **Causes:** NPD crash, misconfiguration, image pull failure, pod eviction
- **Detection gaps:** No alert on NPD absence; no monitoring for missing conditions
- **Action items:** Configure adequate resource requests/limits; implement readiness probes blocking release train; implement monitoring for missing node conditions

#### Failure Mode 3: Over-Sensitive NPD Rules

- **Effects:** Mass node blocking → mass pod drains → significant capacity loss
- **Causes:** Aggressive timeouts, misconfigured thresholds, transient infrastructure false positives
- **Action items:** NRS to maintain tainted node count below configurable threshold; revisit VMSS RP auto-repair batch size; dashboard for individual NPD plugins

#### Failure Mode 4: NPD Health Checker Flapping

- **Effects:** Rapid state changes causing instability, workload disruption from frequent evictions
- **Causes:** Transient network/resource issues, insufficient grace periods
- **Action items:** NRS to add grace periods in tainting logic (multi-stage: T0 → record LTT; T1 → `PreferNoSchedule` + AHE failure; T2 → `NoSchedule`, let watchdog handle)

#### Failure Mode 5: Node Repair Service Replicas Down

- **Effects:** Unhealthy nodes not tainted; healthy nodes not untainted
- **Causes:** Service availability issues, lack of cluster capacity, control plane issues
- **Action items:** Monitor tainted node count; maintain configurable threshold

### 12.3 Node Health Monitoring (NodeHealthAgent)

### 12.6 Kubescape Security Scanning

**Source:** `/docs/kubescape/kubescape-results-publishing-architecture.md`, `/docs/kubescape/kubescape-local-scan-instructions.md`

Kubescape validates Kubernetes clusters against the NSA hardening framework. The architecture consists of two pipelines:

1. **Cluster Deployment Pipeline** (`OneBranch.NonSDP.Cluster.Create.Recycle.yml`): Runs Kubescape scans via EV2 shell extension after cluster deployment. The shell extension downloads the Kubescape binary from GitHub releases, retrieves kubeconfig via idk8sctl, executes the NSA framework scan, and uploads JSON results to Azure Storage using the bootstrapping identity.
2. **Kubescape Publishing Pipeline** (`OneBranch.Dev.Kubescape.yml`, weekly Sunday 3 AM): Downloads JSON results from Azure Storage, processes them via a Python script that splits results by Kubernetes namespace and converts to JUnit XML format with CVSS-aligned severity mapping, then publishes to Azure DevOps test results. Supports both scheduled and manual triggers.

**Design constraint:** OneBranch release pipelines using `pool.type: release` can ONLY execute EV2 deployment tasks. Standard ADO tasks like `PublishTestResults@2` are prohibited, necessitating the two-pipeline architecture.

**Exception management:** Exception files in `src/tools/kubescape/exceptions/` are merged via `jq -s 'add'` before scanning. Local scans follow the documented instructions:

```bash
jq -s 'add' src/tools/kubescape/exceptions/*.json > kubescape-exceptions-merged.json
kubescape scan framework nsa \
  --kubeconfig path/to/kubeconfig \
  --exceptions kubescape-exceptions-merged.json \
  --format pretty-printer --verbose
```

**Current Kubescape version at time of writing:** v3.0.42

### 12.7 Scheduled Event Handling

Per ADR-0012, scheduled event approval follows a coordinated criteria:
1. Node must have the `maintenance.k8s.msidentity.io/scheduled-event` taint (set by NRS)
2. Node must have no workload pods running (only static/DaemonSet pods remain)

Both conditions must be satisfied before an event is approved. This balances workload stability (avoids PDB violations and data loss) with maintenance speed.

### 14.3 Migration Plans

#### CRD Migration (MVP Phase 3)

The Management Plane currently stores state in Kubernetes ConfigMaps via `IPersistentStore`. The plan is to migrate to Custom Resource Definitions (CRDs) for structured, versioned, observable state. This would enable:
- Kubernetes-native watch/list semantics
- Typed schema validation
- Integration with operator patterns (reconciliation loops)

**Gap identified:** No ADR exists yet for this migration. Recommendation: author ADR-0013 covering CRD schema design, migration strategy from ConfigMaps, and rollback plan.

#### dSTS Adoption

Multiple docs reference the shift from Entra → dSTS authentication:
- **SP OBO flow:** EV2 passes a deployment security group identity → dSTS creates a delegated compound token scoped to specific K8s namespaces
- **User OBO flow:** EV2 passes the approver's dSTS identity → fine-grained per-user audit trail
- **Impact:** Would enable proper tenant isolation at the K8s API level (currently both `workload-provisioner` and `workload-deployer` roles bound to same MP SP)

#### Workload Identity

The MP currently uses Node MSI to access Azure Key Vault for the SN+I certificate. Per documentation, "This solution should be replaced with native dSMS support." The migration to DSMS/dSTS-based identity (automated per ADR-0010) is ongoing.

#### Component Deployer HTTP Extension Migration

The Component Deployer currently runs as an EV2 **shell extension** (`ComponentDeployer.CLI`). The code is explicitly split into two projects (`ComponentDeployer.CLI` + `ComponentDeployer` core library) to prepare for future migration to an EV2 **HTTP extension**, which would provide better observability and integration with EV2's management plane.
