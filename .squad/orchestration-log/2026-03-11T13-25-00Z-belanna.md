## Assignment

Investigate and solve Azure DevOps multi-org access issue: Squad agents need to access repos and PRs across multiple organizations (`microsoft` and `msazure`), but MCP tools are single-org by design.

**Blocking Issue:** PR #15000967 in msazure/CESEC/CIEng-Infra-AKS could not be reviewed because MCP was locked to `microsoft` org.

---

## Work Completed

### 1. Root Cause Analysis
- Identified that `@azure-devops/mcp` package requires org name as startup argument
- Confirmed no runtime reconfiguration capability exists
- Validated that repo-level config doesn't override global MCP server instances
- Documented auth method: Entra ID interactive (PATs not supported)

### 2. Solution Design
**Multi-instance MCP pattern** with separate named server instances:
```json
"ado-microsoft": { "args": ["@azure-devops/mcp", "microsoft"] }
"ado-msazure": { "args": ["@azure-devops/mcp", "msazure"] }
```
- Tools become prefixed: `ado-microsoft-core_list_projects` vs `ado-msazure-core_list_projects`
- Both orgs accessible simultaneously without context switching

### 3. Alternatives Evaluated & Rejected
- **Az CLI fallback** (slower, less structured, failing in test environment)
- **Dynamic config swapping** (no runtime reload without session restart)

### 4. Decision Documentation
- Wrote comprehensive decision document: `.squad/decisions/inbox/belanna-ado-multiorg.md`
- Included implementation phases (Config → Validation → Documentation → Optional auto-detection)
- Documented trade-offs and success metrics

### 5. Knowledge Capture
- Updated `.squad/agents/belanna/history.md` with technical learnings
- Documented solution and recommendation for future cross-org work
