## Overview

This document defines the complete deployment plan for Squad on DK8S — Microsoft's internal Azure Kubernetes platform. It covers the DK8S-specific onboarding path, ConfigGen integration, ArgoCD structure, identity, secrets, node pool placement, EV2 packaging, and observability. It is the execution blueprint for the acceptance criteria in issue #1061.

DK8S is **not** generic AKS. It has:
- **ConfigGen** — typed C# configuration generation, not raw YAML/Helm values
- **ArgoCD + GitOps** — all deployments via app-of-apps; no `kubectl apply`
- **EV2** — safe deployment rollout, canary staging, health monitor integration
- **Workload Identity** — Azure AD federated credentials, no stored service principal secrets
- **DK8S RBAC** — platform-managed namespace and service account provisioning

## 1. DK8S Onboarding Requirements

### 1.1 Tenant Registration

Every workload on DK8S must be registered as a **tenant** in the DK8S inventory. For Squad:

```yaml
# inventory/tenants/squad.yaml
tenant:
  name: squad-hq
  owner: tamirdresher@microsoft.com
  team: dk8s-squad
  tier: internal-tooling
  description: |
    Squad multi-agent AI framework. Provides autonomous agent orchestration
    for DK8S swimlane teams. Ralph (orchestrator) runs as a CronJob every 5 minutes;
    specialist agents run as on-demand Kubernetes Jobs.
  contacts:
    - alias: tamirdresher
      role: owner
    - alias: dk8s-platform
      role: oncall
  sla: best-effort
  environment: nonprod    # Phase 1; upgrade to prod after 30-day stability
```

**Action:** Submit tenant registration PR to `microsoft-mtp/dk8s-inventory`.

### 1.2 Namespace

Squad gets its own namespace for isolation. One namespace per swimlane Squad deployment (not shared), per the ADR-001 topology decision.

```yaml
# Namespace manifest (generated by ConfigGen — do not hand-edit)
apiVersion: v1
kind: Namespace
metadata:
  name: squad-hq
  labels:
    dk8s.microsoft.com/tenant: squad-hq
    dk8s.microsoft.com/tier: internal-tooling
    dk8s.microsoft.com/managed-by: argocd
  annotations:
    dk8s.microsoft.com/owner: tamirdresher@microsoft.com
    dk8s.microsoft.com/oncall: dk8s-squad
```

**Naming convention:**
- Org Squad: `squad-org`
- Swimlane Squad: `squad-{swimlane}` (e.g., `squad-fedramp`, `squad-infra`)
- Personal Squad: Not deployed to DK8S — runs locally only

### 1.3 Service Account

Each Squad instance gets a Kubernetes service account federated to Azure AD for Workload Identity:

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: squad-hq-sa
  namespace: squad-hq
  annotations:
    # Workload Identity: binds to Azure Managed Identity
    azure.workload.identity/client-id: "${SQUAD_MANAGED_IDENTITY_CLIENT_ID}"
    azure.workload.identity/tenant-id: "${AZURE_TENANT_ID}"
  labels:
    azure.workload.identity/use: "true"
```

**RBAC for the service account:**

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: squad-hq-role
  namespace: squad-hq

  # Ralph needs to create/watch Jobs for agent dispatch

  # Agents need ConfigMaps for state/coordination
  - apiGroups: [""]
    resources: ["configmaps", "secrets"]
    verbs: ["get", "list", "watch", "create", "update", "patch"]
  # Pod logs for diagnostics
  - apiGroups: [""]
    resources: ["pods", "pods/log"]
    verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: squad-hq-rolebinding
  namespace: squad-hq
subjects:
  - kind: ServiceAccount
    name: squad-hq-sa
    namespace: squad-hq
roleRef:
  kind: Role
  name: squad-hq-role
```

### 2.4 Ralph Orchestrator Config (`RalphConfig.cs`)

```csharp
using ConfigurationGeneration.Dk8sApplication;
using ConfigurationGeneration.WorkloadIdentity;

/// <summary>
/// Ralph runs as a CronJob. Every 5 minutes it:
///   1. Reads .squad/pending-issues/
///   2. Reads the workqueue ConfigMap
///   3. Dispatches specialist agent Jobs
///   4. Updates circuit breaker state
/// </summary>
public class RalphConfig : IDk8sComponent
{
    public string Name => "ralph-orchestrator";
    public string Kind => "CronJob";

    public CronJobSpec Spec => new()
    {
        Schedule = "*/5 * * * *",
        TimeZone = "UTC",
        ConcurrencyPolicy = ConcurrencyPolicy.Forbid,   // Never overlap Ralph runs
        SuccessfulJobsHistoryLimit = 3,
        FailedJobsHistoryLimit = 5,
        StartingDeadlineSeconds = 60,
        JobTemplate = new JobTemplateSpec
        {
            ActiveDeadlineSeconds = 270,   // 4.5 min — must finish before next run
            BackoffLimit = 0,              // No retry; next cron picks up
            Container = new ContainerSpec
            {
                Name = "ralph",
                Image = "mcr.microsoft.com/dk8s/squad/ralph:latest",
                ImagePullPolicy = ImagePullPolicy.Always,
                Command = ["pwsh", "-File", "/app/ralph-watch.ps1"],
                Resources = new ResourceRequirements
                {
                    Requests = new() { Cpu = "100m", Memory = "256Mi" },
                    Limits   = new() { Cpu = "500m", Memory = "512Mi" },
                },
                EnvFrom = [new EnvFromSource { SecretRef = "squad-secrets" }],
                VolumeMounts =
                [
                    new() { Name = "squad-state", MountPath = "/app/.squad" },
                    new() { Name = "git-cache",   MountPath = "/app/.git-cache" },
                ],
            },
            Volumes =
            [
                new() { Name = "squad-state", PersistentVolumeClaim = "squad-state-pvc" },
                new() { Name = "git-cache",   EmptyDir = new() },
            ],
            ServiceAccountName = "squad-hq-sa",
            NodeSelector = new() { ["dk8s.microsoft.com/pool"] = "user" },
        },
    };
}
```

### 2.5 Agent Job Config (`AgentJobConfig.cs`)

Specialist agents (Picard, Data, Belanna, etc.) are dispatched as on-demand Jobs by Ralph:

```csharp
using ConfigurationGeneration.Dk8sApplication;

/// <summary>
/// Template for agent Jobs dispatched by Ralph at runtime.
/// Ralph creates a Job from this template per-agent-dispatch.
/// </summary>
public class AgentJobConfig : IDk8sComponent
{
    public string Name => "agent-job-template";

    // This generates a ConfigMap that Ralph uses to create Jobs dynamically
    public ConfigMapData JobTemplate => new()
    {
        ["template"] = """
        apiVersion: batch/v1
        kind: Job
        metadata:
          generateName: squad-agent-
          namespace: squad-hq
          labels:
            dk8s.microsoft.com/component: squad-agent
            squad.microsoft.com/agent: "{{AGENT_NAME}}"
            squad.microsoft.com/issue: "{{ISSUE_NUMBER}}"
        spec:
          ttlSecondsAfterFinished: 3600
          activeDeadlineSeconds: 1800
          backoffLimit: 1
          template:
            spec:
              serviceAccountName: squad-hq-sa
              restartPolicy: OnFailure
              nodeSelector:
                dk8s.microsoft.com/pool: user
              containers:
                - name: agent
                  image: mcr.microsoft.com/dk8s/squad/agent:latest
                  command: ["gh", "copilot", "suggest", "--agent", "{{AGENT_NAME}}"]
                  resources:
                    requests: { cpu: "250m", memory: "512Mi" }
                    limits:   { cpu: "2",    memory: "2Gi"  }
                  envFrom:
                    - secretRef:
                        name: squad-secrets
                  volumeMounts:
                    - name: squad-state
                      mountPath: /app/.squad
              volumes:
                - name: squad-state
                  persistentVolumeClaim:
                    claimName: squad-state-pvc
                    readOnly: false
        """
    };
}
```

### 2.6 squad.config.ts ↔ ConfigGen Mapping

```
squad.config.ts (Squad layer)          ConfigGen (DK8S layer)
─────────────────────────────          ──────────────────────
models.defaultModel           →        SquadRuntimeConfig.DefaultModel (env var)
models.fallbackChains         →        SquadRuntimeConfig.FallbackChain (env var)
routing.rules                 →        ConfigMap: squad-routing-config
casting.allowlistUniverses    →        ConfigMap: squad-casting-config
agents (charters)             →        Agent Job template labels
schedule                      →        CronJob schedule
```

ConfigGen **generates the infrastructure**; `squad.config.ts` **governs agent behavior**. They are complementary layers.

## 4. Identity — Workload Identity & GitHub Federation

### 4.1 Azure Managed Identity

Each Squad deployment gets a User-Assigned Managed Identity (UAMI):

```bash
# Provision via Azure CLI (or EV2 provisioning step)
az identity create \
  --name squad-hq-identity \
  --resource-group dk8s-squad-rg \
  --location eastus

# Get client ID for ConfigGen annotation
az identity show \
  --name squad-hq-identity \
  --resource-group dk8s-squad-rg \
  --query clientId -o tsv
```

### 4.2 Federated Credential for Kubernetes

```bash
# Federate the UAMI to the Kubernetes service account
az identity federated-credential create \
  --name squad-hq-k8s-fedcred \
  --identity-name squad-hq-identity \
  --resource-group dk8s-squad-rg \
  --issuer "${AKS_OIDC_ISSUER}" \
  --subject "system:serviceaccount:squad-hq:squad-hq-sa" \
  --audience api://AzureADTokenExchange
```

### 4.3 GitHub Token Federation

Squad agents need GitHub access (read issues, create PRs, manage projects). Use GitHub's OIDC provider to federate — no stored PATs:

```bash
# Federate the UAMI to GitHub Actions OIDC (for CI-triggered agents)
az identity federated-credential create \
  --name squad-hq-github-fedcred \
  --identity-name squad-hq-identity \
  --resource-group dk8s-squad-rg \
  --issuer "https://token.actions.githubusercontent.com" \
  --subject "repo:microsoft-mtp/dk8s-squad:ref:refs/heads/main" \
  --audience api://AzureADTokenExchange
```

For **runtime** GitHub access (Ralph dispatching agents to work GitHub issues), the GitHub App installation token is stored in Key Vault and accessed via Workload Identity (see Section 5). No service account tokens or PATs are stored in Kubernetes secrets unencrypted.

### 4.4 gh CLI Authentication in Pods

Agents authenticate to `gh` CLI using a GitHub App installation token fetched at startup:

```powershell
# Startup script in agent container (ralph-watch.ps1 / agent entrypoint)
# 1. Exchange K8s service account token for Azure AD token (automatic with Workload Identity)

# 2. Fetch GitHub App installation token from Key Vault
$GH_TOKEN = az keyvault secret show `
  --vault-name squad-hq-kv `
  --name github-app-installation-token `
  --query value -o tsv

# 3. Authenticate gh CLI
$env:GH_TOKEN = $GH_TOKEN
gh auth status  # Verify
```

The Key Vault managed identity access is configured via RBAC (Section 5.2).

### 5.1 Key Vault Inventory

```
├── Secrets:
```

### 5.2 Key Vault Access Policy

```bash
# Grant Squad's managed identity access to Key Vault secrets
az keyvault set-policy \
  --name squad-hq-kv \
  --object-id $(az identity show --name squad-hq-identity \
    --resource-group dk8s-squad-rg --query principalId -o tsv) \
```

### 5.3 Secret Injection via CSI Driver

DK8S uses the Azure Key Vault CSI driver to mount secrets as files (not env vars — reduces secret exposure in process listings):

```yaml
# SecretProviderClass (generated by ConfigGen)
apiVersion: secrets-store.csi.x-k8s.io/v1
kind: SecretProviderClass
metadata:
  name: squad-hq-secrets
  namespace: squad-hq
spec:
  provider: azure
  secretObjects:
    # Sync KV secrets to K8s Secret for envFrom usage
    - secretName: squad-secrets
      type: Opaque
      data:
        - objectName: github-app-installation-token
          key: GH_TOKEN
        - objectName: teams-webhook-url
          key: TEAMS_WEBHOOK_URL
        - objectName: anthropic-api-key
          key: ANTHROPIC_API_KEY
        - objectName: openai-api-key
          key: OPENAI_API_KEY
  parameters:
    usePodIdentity: "false"
    useVMManagedIdentity: "false"
    clientID: "${SQUAD_MANAGED_IDENTITY_CLIENT_ID}"
    keyvaultName: squad-hq-kv
    tenantId: "${AZURE_TENANT_ID}"
    objects: |
      array:
        - |
          objectName: github-app-installation-token
          objectType: secret
        - |
          objectName: teams-webhook-url
          objectType: secret
        - |
          objectName: anthropic-api-key
          objectType: secret
        - |
          objectName: openai-api-key
          objectType: secret
```

### 5.4 Token Rotation

GitHub App installation tokens expire after 1 hour. A rotation CronJob renews them:

```yaml
# Rotation CronJob — every 50 minutes
apiVersion: batch/v1
kind: CronJob
metadata:
  name: squad-token-refresh
  namespace: squad-hq
spec:
  schedule: "*/50 * * * *"
  jobTemplate:
    spec:
      template:
        spec:
          serviceAccountName: squad-hq-sa
          restartPolicy: OnFailure
          containers:
            - name: token-refresh
              image: mcr.microsoft.com/dk8s/squad/token-refresh:latest
              command:
                - pwsh
                - -Command
                - |
                  # Generate new installation token using GitHub App private key
                  $jwt = New-GitHubAppJWT -AppId $env:GITHUB_APP_ID -PrivateKey $env:GITHUB_APP_PRIVATE_KEY
                  $token = Get-GitHubInstallationToken -JWT $jwt -InstallationId $env:GITHUB_APP_INSTALLATION_ID
                  # Store in Key Vault (overwrites previous)
                  az keyvault secret set --vault-name squad-hq-kv --name github-app-installation-token --value $token
              envFrom:
                - secretRef:
                    name: squad-secrets
```

## 6. Node Pool Assignment

### 6.1 DK8S Node Pools

DK8S clusters have multiple node pools with different characteristics:

| Pool | Purpose | Machine SKU | Outbound Internet |
|------|---------|-------------|-------------------|
| `system` | Platform workloads, low-latency ops | Standard_D2s_v3 | No (internal only) |
| `user` | Application workloads | Standard_D4s_v3 | Yes (GitHub API) |
| `gpu` | ML inference | Standard_NC* | Yes |
| `spot` | Batch/bursty workloads | Spot SKU | Yes |

### 6.2 Squad Placement Rationale

| Component | Pool | Reason |
|-----------|------|--------|
| Ralph CronJob | `user` | Needs GitHub API access; moderate compute |
| Agent Jobs | `user` | Need GitHub API access; bursty compute |
| Token Refresh CronJob | `user` | Needs GitHub API + Key Vault access |
| MCP Server Deployment | `system` | Internal-only; low latency to other pods |

```yaml
# NodeSelector for Ralph and Agent workloads
nodeSelector:
  dk8s.microsoft.com/pool: user
  kubernetes.io/os: linux

# Toleration for user pool (if tainted)
tolerations:
  - key: "dk8s.microsoft.com/pool"
    operator: "Equal"
    value: "user"
    effect: "NoSchedule"
```

### 6.3 Resource Sizing

Based on Squad's current workload profile (5-minute polling, max 3 concurrent agents):

| Component | CPU Request | CPU Limit | Mem Request | Mem Limit |
|-----------|------------|-----------|------------|-----------|
| Ralph CronJob | 100m | 500m | 256Mi | 512Mi |
| Agent Job | 250m | 2000m | 512Mi | 2Gi |
| Token Refresh | 50m | 100m | 64Mi | 128Mi |
| MCP Server | 100m | 500m | 256Mi | 512Mi |

## 8. Monitoring — Azure Monitor & DK8S Observability

### 8.1 Metrics

Squad exposes metrics via a Prometheus-compatible `/metrics` endpoint on the MCP server. DK8S's built-in Prometheus scrapes these automatically.

**Key metrics to instrument:**

| Metric | Type | Description |
|--------|------|-------------|
| `squad_ralph_runs_total` | Counter | Total Ralph CronJob executions |
| `squad_ralph_issues_processed_total` | Counter | Issues processed per run |
| `squad_agent_jobs_dispatched_total` | Counter | Agent Jobs created |
| `squad_agent_jobs_succeeded_total` | Counter | Agent Jobs succeeded |
| `squad_agent_jobs_failed_total` | Counter | Agent Jobs failed |
| `squad_agent_job_duration_seconds` | Histogram | Agent Job duration |
| `squad_circuit_breaker_state` | Gauge | 0=closed, 1=open |
| `squad_github_token_expiry_seconds` | Gauge | Time until token expiry |

### 8.2 Prometheus Alert Rules

```yaml
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: squad-hq-alerts
  namespace: squad-hq
spec:
  groups:
    - name: squad.health
      interval: 1m
      rules:
        - alert: RalphNotRunning
          expr: |
            time() - max(kube_cronjob_status_last_schedule_time{cronjob="ralph-orchestrator"}) > 600
          for: 5m
          labels:
            severity: warning
            component: squad-ralph
          annotations:
            summary: "Ralph CronJob hasn't run in 10 minutes"
            runbook: "https://eng.ms/docs/dk8s/squad/runbooks/ralph-not-running"

        - alert: HighAgentFailureRate
          expr: |
            rate(squad_agent_jobs_failed_total[15m])
            / rate(squad_agent_jobs_dispatched_total[15m]) > 0.5
          for: 5m
          labels:
            severity: critical
            component: squad-agents
          annotations:
            summary: "More than 50% of agent jobs are failing"

        - alert: CircuitBreakerOpen
          expr: squad_circuit_breaker_state == 1
          for: 1m
          labels:
            severity: warning
          annotations:
            summary: "Squad circuit breaker is open — agent dispatch halted"

        - alert: GitHubTokenExpiringSoon
          expr: squad_github_token_expiry_seconds < 300
          labels:
            severity: warning
          annotations:
            summary: "Squad GitHub token expires in less than 5 minutes"
```

### 8.4 Log Analytics Query Examples

```kql
// Ralph failures in last 30 minutes
ContainerLog
| where ContainerName == "ralph"
| where LogMessage contains "ERROR" or LogMessage contains "FATAL"
| where TimeGenerated > ago(30m)
| summarize count() by bin(TimeGenerated, 5m), LogMessage
| where count_ > 3

// Agent job duration percentiles
ContainerLog
| where ContainerName startswith "squad-agent-"
| where LogMessage contains "completed in"
| parse LogMessage with * "completed in " duration:double " seconds"
| summarize p50=percentile(duration, 50), p95=percentile(duration, 95), p99=percentile(duration, 99) by bin(TimeGenerated, 1h)
```

## 9. Deployment Decision: Single-Tenant vs. Multi-Tenant

**Decision: Single-tenant per swimlane namespace** (per ADR-001 topology).

| Option | Verdict | Rationale |
|--------|---------|-----------|
| One Squad instance serves all swimlanes | ❌ Rejected | Agent state isolation is critical; a rogue agent in one swimlane must not affect others |
| One instance per swimlane namespace | ✅ Adopted | Clean isolation; aligns with ADR-001 three-tier hierarchy; ArgoCD ApplicationSet makes this lightweight |
| Operator with SquadAgent CRDs | 🔮 Future (Phase 3 per #1039) | Start with Helm; graduate to operator after stability proven |

Each swimlane gets its own: namespace, managed identity, Key Vault access, ArgoCD Application, PVC for `.squad/` state.

Shared across swimlanes: container images, Grafana instance, GitHub App (multiple installation tokens).

## 10. Step-by-Step Onboarding Checklist

### Phase 0: Prerequisites (Platform Team)

- [ ] **P0.1** DK8S cluster with Workload Identity enabled (`--enable-oidc-issuer --enable-workload-identity`)
- [ ] **P0.2** Azure Key Vault CSI driver installed on cluster
- [ ] **P0.3** ArgoCD instance running, project `dk8s-internal-tooling` exists
- [ ] **P0.4** Prometheus + Grafana stack deployed (DK8S standard monitoring)
- [ ] **P0.5** `mcr.microsoft.com/dk8s/squad/*` container images published

### Phase 1: Tenant Registration (Squad Owner)

- [ ] **1.1** Create tenant registration PR → `microsoft-mtp/dk8s-inventory`: `inventory/tenants/squad.yaml`
- [ ] **1.2** Get tenant approved by DK8S platform team
- [ ] **1.3** Note the assigned cluster name and OIDC issuer URL

### Phase 2: Azure Resources (Squad Owner + Platform Team)

- [ ] **2.1** Create Resource Group: `az group create --name dk8s-squad-rg --location eastus`
- [ ] **2.2** Create User-Assigned Managed Identity: `az identity create --name squad-hq-identity --resource-group dk8s-squad-rg`
- [ ] **2.3** Create Key Vault: `az keyvault create --name squad-hq-kv --resource-group dk8s-squad-rg`
- [ ] **2.4** Store initial secrets in Key Vault (GitHub App ID, private key, Teams webhook, API keys)
- [ ] **2.5** Grant UAMI access to Key Vault: `az keyvault set-policy --name squad-hq-kv --object-id <uami-principal-id> --secret-permissions get list`
- [ ] **2.6** Create federated credential (K8s → Azure AD)
- [ ] **2.7** Create federated credential (GitHub OIDC → Azure AD)

### Phase 3: ConfigGen (Squad Owner)

- [ ] **3.1** Clone `microsoft-mtp/dk8s-squad`
- [ ] **3.2** Add `src/Squad.ConfigGen/` project (see Section 2)
- [ ] **3.3** Fill in UAMI client ID and Key Vault name in `SquadConfig.cs`
- [ ] **3.4** Run ConfigGen locally: `dotnet run --project src/Squad.ConfigGen -- --output out/`
- [ ] **3.5** Verify generated manifests in `out/`
- [ ] **3.6** Commit generated manifests to `argocd/manifests/`

### Phase 4: ArgoCD Registration (Platform Team)

- [ ] **4.1** Add squad-hq Application to ArgoCD app-of-apps repo
- [ ] **4.2** Verify ArgoCD sync: `argocd app sync squad-hq`
- [ ] **4.3** Check all resources `Synced` + `Healthy` in ArgoCD UI
- [ ] **4.4** Add swimlanes to ApplicationSet as needed

### Phase 5: Namespace Bootstrap

- [ ] **5.1** Verify namespace created: `kubectl get ns squad-hq`
- [ ] **5.2** Verify service account with UAMI annotation: `kubectl get sa squad-hq-sa -n squad-hq -o yaml`
- [ ] **5.3** Verify SecretProviderClass: `kubectl get secretproviderclass -n squad-hq`
- [ ] **5.4** Verify `squad-secrets` K8s Secret populated: `kubectl get secret squad-secrets -n squad-hq`
- [ ] **5.5** Verify PVC bound: `kubectl get pvc squad-state-pvc -n squad-hq`

### Phase 6: Smoke Test

- [ ] **6.1** Trigger Ralph CronJob manually: `kubectl create job --from=cronjob/ralph-orchestrator ralph-smoke-test -n squad-hq`
- [ ] **6.2** Check Ralph logs: `kubectl logs job/ralph-smoke-test -n squad-hq -f`
- [ ] **6.3** Verify Ralph completes without error (exit 0)
- [ ] **6.4** Create a test GitHub issue with `squad` label; verify Ralph picks it up and dispatches an agent Job
- [ ] **6.5** Verify metrics endpoint reachable: `kubectl port-forward svc/squad-mcp-server 8080:8080 -n squad-hq` → `curl localhost:8080/metrics`

### Phase 7: EV2 Pipeline (Platform Team + Squad Owner)

- [ ] **7.1** Register Squad as EV2 service: add `ev2/ServiceModel.json`
- [ ] **7.2** Configure OneBranch pipeline (`.pipelines/squad-release.yml`)
- [ ] **7.3** Run pipeline: verify ConfigGen runs, Helm chart packages, EV2 deploys to canary
- [ ] **7.4** After 30-minute bake on canary, verify health monitors pass
- [ ] **7.5** EV2 promotes to production

### Phase 8: Observability Validation
