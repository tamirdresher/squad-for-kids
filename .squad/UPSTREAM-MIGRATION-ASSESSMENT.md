## 1. What Upstream Is Changing and Why

### Source
- **Discussion:** [bradygaster/squad#499](https://github.com/bradygaster/squad/discussions/499)
- **PRD:** [bradygaster/squad#498](https://github.com/bradygaster/squad/issues/498)
- **Announced:** ~2026-03-22, by @bradygaster

### The Change
The upstream Squad framework will **remove `.squad/` from version control entirely** before v1.0. The directory currently exists in the upstream repo with ~128 tracked files, ~889 KB.

**Rationale (from PRD #498):**
- SDK mode (`squad.config.ts` + `squad build`) already generates `.squad/` as **build output**
- Tracking build output in git causes: merge conflicts on append-only files, dirty working trees when agents work across branches, CI noise, privacy leaks (session logs in public repos), conceptual confusion
- `.squad/` is **runtime state**, not source code

### Timeline
| Milestone | When | What Happens |
|-----------|------|--------------|
| **v0.9.0** | "This weekend" (~2026-03-22) | Announce plan, publish migration docs, begin `squad export` tooling |
| **v0.10.0** | Next release after v0.9.0 | `.squad/` added to upstream `.gitignore`, untracked with `git rm --cached` |
| **v1.0** | Future | `.squad/` is fully local-only, generated by `squad build` |

**Window:** ~1 full release cycle between v0.9.0 and v0.10.0.

### New Migration Commands (coming in v0.9.0)
- `squad export` — saves current team state (charters, history, decisions) to portable format
- `squad build` — regenerates `.squad/` from `squad.config.ts`
- `squad init --sdk` — initializes SDK mode if not already adopted

## 4. What Will Be Affected

### The Core Question
The upstream change is specifically about removing `.squad/` from **the upstream framework repo's** version control. For **user repos** (like ours), the migration is:
1. Decide whether to keep `.squad/` tracked OR untrack it
2. If untracking: ensure `squad build` regenerates everything needed

### Our Situation: Hybrid Architecture
Our `.squad/` contains **two distinct categories**:

#### Category A: Framework-Generated (can be regenerated by `squad build`)
- Agent charters and templates derived from `squad.config.ts`
- Routing, casting, skill registrations
- Standard squad infrastructure

#### Category B: Tamir-Specific Institutional Knowledge (NOT regeneratable)
These are **irreplaceable** and must be preserved explicitly:
- **860 log/orchestration entries** (already gitignored, no action needed)
- **Agent histories** (`agents/*/history.md`, `agents/*/history-2026-Q1.md`) — accumulated team memory
- **Decisions** (`decisions.md`, 46 active decisions) — team governance
- **Custom skills** (44 skills, domain-specific to this project/team)
- **Cross-machine state** — coordination configs
- **Scripts** — automation tooling specific to tamresearch1
- **Research artifacts** — institutional knowledge

### Phase 2: Decision Point (When v0.9.0 Ships)
**Strategic choice required:**

#### Option A: Keep `.squad/` Tracked (Recommended for us)
- **Rationale:** Our `.squad/` contains irreplaceable institutional knowledge (860 files of history, 44 custom skills, 46 decisions, custom agent configurations) that is NOT regeneratable from `squad.config.ts`
- **Action:** Do nothing to version control. Monitor upstream v0.10.0 changes. Optionally improve `.gitignore` tiering.
- **Risk:** We diverge from upstream default behavior. Framework upgrades may become more complex.
- **Mitigation:** This is an **explicit opt-in** per upstream (see PRD: "Users without SDK mode will need to either adopt SDK mode or keep `.squad/` tracked — opt-in, not the default")

#### Option B: Untrack `.squad/`, Move Knowledge to Source
- **Rationale:** Follow upstream direction, reduce merge conflicts
- **Action:** 
  1. Extract all irreplaceable content to source files (decisions → `TEAM_DECISIONS.md`, histories → archived markdown files, skills → tracked separately)
  2. Add `.squad/` to `.gitignore`
  3. Run `git rm -r --cached .squad/`
  4. Ensure CI runs `squad build` before any step that reads `.squad/`
- **Risk:** Significant migration effort. Loss of historical context if not carefully extracted. CI pipeline changes required.

### Phase 3: v0.10.0 Preparation (Before Upstream Release)
1. Watch upstream repo for `squad export` command availability
2. Test `squad build` regeneration: does it correctly rebuild from our `squad.config.ts`?
3. Verify CI workflows don't break if `.squad/` isn't pre-committed

## 7. Recommendation

**Adopt Option A (keep `.squad/` tracked)** for the following reasons:

1. **Our custom skills are not regeneratable** — 44 skills with detailed domain-specific implementations are stored as markdown/scripts in `.squad/skills/`. These have no representation in `squad.config.ts` and cannot be regenerated by `squad build`.

2. **Agent history is institutional knowledge** — 17 active agents with accumulated histories representing months of team learning. This is not build output; it's organizational memory.

3. **Upstream explicitly supports opt-in tracking** — PRD #498 states: "Users without SDK mode will want to adopt it OR keep `.squad/` tracked (opt-in, not the default)."

4. **We're already using SDK mode** — `squad.config.ts` exists and is fully configured. We can always run `squad build` to regenerate the framework parts.

5. **Our `.gitignore` tiering is already good** — 330+ runtime files are already excluded. The tracked content is intentional.
