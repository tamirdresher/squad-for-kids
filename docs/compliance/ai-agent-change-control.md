# AI Agent Change Control Policy

## 1. Purpose and Scope

This policy establishes formal governance over code and documentation changes generated by AI agents operating within the Squad framework. It ensures that all AI-generated contributions meet the same quality, security, and auditability standards as human-authored changes.

**Scope:** All repositories managed by Squad teams where AI agents (e.g., GitHub Copilot, squad-assigned agents) propose, create, or modify code, configuration, documentation, or infrastructure artifacts.

## 2. Change Classification

Every AI-generated change must be classified before it enters the review pipeline.

### 2.1 Routine (Low-Risk)

| Criteria | Examples |
|----------|----------|
| Documentation-only changes | README updates, inline comment fixes, markdown formatting |
| Dependency version bumps (patch) | `1.2.3` → `1.2.4` with passing CI |
| Test additions with no production code changes | New unit tests, snapshot updates |
| Linting / formatting fixes | Auto-fix from ESLint, Prettier, or equivalent |

### 2.2 Standard (Medium-Risk)

| Criteria | Examples |
|----------|----------|
| Bug fixes with clear reproduction steps | Null-check additions, off-by-one corrections |
| Small feature work with defined specs | Adding a new API field, UI label change |
| Configuration changes | Environment variable additions, CI workflow edits |

### 2.3 Critical (High-Risk)

| Criteria | Examples |
|----------|----------|
| Security-sensitive changes | Auth logic, encryption, secrets handling |
| Architecture or design changes | New service boundaries, database schema migration |
| Infrastructure / deployment changes | Terraform, Helm charts, production pipelines |
| Changes affecting data handling or PII | Logging changes, data export features |

## 3. Review Requirements by Change Type

### 3.1 Routine Changes

- **Minimum reviewers:** 0 (auto-merge eligible)
- **CI requirement:** All checks must pass
- **Auto-merge:** Permitted when CI is green and the change is classified as Routine
- **Post-merge audit:** Included in weekly change summary

### 3.2 Standard Changes

- **Minimum reviewers:** 1 squad member with domain expertise
- **CI requirement:** All checks must pass
- **Auto-merge:** Not permitted; requires explicit approval
- **Review SLA:** Within 1 business day

### 3.3 Critical Changes

- **Minimum reviewers:** 2 (including squad lead or security delegate)
- **CI requirement:** All checks must pass, including security scans
- **Auto-merge:** Not permitted
- **Additional gate:** Squad lead (@picard) must approve
- **Review SLA:** Within 4 hours for security issues; 1 business day otherwise

## 4. Audit Trail Requirements

All AI-generated changes must maintain a complete, tamper-evident audit trail.

### 4.1 Git History

- Every commit must include a `Co-authored-by` trailer identifying the AI agent
- Commit messages must reference the originating issue or task (e.g., `#401`)
- Force-pushes to shared branches are prohibited

### 4.2 Pull Request Documentation

- PR description must link the originating issue with `Closes #<number>`
- AI agent actions must be visible in PR comments or timeline
- Classification rationale must be stated in the PR body or a comment

### 4.3 Issue Linking

- Every PR must be linked to at least one issue or work item
- Issue must document the intent, acceptance criteria, and risk classification
- Completed work must update the issue status

### 4.4 Retention

- Git history: Retained indefinitely
- PR comments and reviews: Retained for the life of the repository
- CI/CD logs: Retained per organizational log-retention policy (minimum 1 year)

## 5. Rollback Procedures

### 5.1 Standard Rollback

1. Identify the merge commit via `git log --merges`
2. Open a revert PR: `git revert -m 1 <merge-commit-sha>`
3. Revert PR follows the same review requirements as the original change classification
4. Notify the squad channel with a link to the revert PR

### 5.2 Expedited Rollback

When a merged change causes a production incident:

1. Any squad member may open and self-merge a revert PR
2. CI must still pass on the revert
3. A post-incident note must be added to the original issue within 24 hours
4. Squad lead reviews the incident in the next retrospective

## 6. Emergency Change Process

For changes required outside normal review flows (e.g., active security incident):

1. **Authorization:** Squad lead or designated security delegate must verbally or in writing approve the emergency change
2. **Execution:** Change is committed directly to a protected branch or merged with bypassed branch protection, with a bypass reason documented
3. **Documentation:** Within 24 hours, the author must:
   - Create a retrospective issue describing the emergency
   - Link the commit(s) to the retrospective issue
   - Classify the change retroactively
4. **Review:** The emergency change must receive a full review within 3 business days
5. **Tracking:** Emergency changes are flagged in the quarterly compliance report
