## What This Area Does

The infrastructure area owns all cluster provisioning, deployment automation, and platform
operations for the tamresearch1 system. It covers:

- **AKS** (Azure Kubernetes Service) cluster lifecycle — automatic provisioning via Bicep
- **Helm charts** — `squad` and `squad-agents` chart packages for K8s deployment
- **KEDA** (Kubernetes Event-Driven Autoscaling) configuration
- **Gateway API** — ingress routing via Nginx and Gateway API resources
- **Docker** — container definitions and compose files for local dev
- **Environments** — parameter files for dev / stg / ppe / stg-gov / prod rings
- **AZD AI Agent integration** — Azure Developer CLI infra for agent deployment
- **Deployment scripts** — PowerShell and Bash deploy helpers (`deploy-*.ps1`, `deploy-*.sh`)
- **Alerting** — Bicep templates for Azure Monitor alert rules and cache alerts

Changes here have **broad blast radius** — they can affect all running agents and services.
Treat with care; prefer incremental rollouts.

## Constraints

- Never commit real credentials or secrets — use Azure Key Vault references or GitHub Secrets
- Prod ring parameters (`environments/prod.parameters.json`) require explicit Picard approval before any apply
- All new Bicep modules must be verified against the active AKS automatic tier API version
- KEDA ScaledObjects must include `minReplicaCount` to avoid zero-replica failures in prod
- AZD AI Agent infrastructure lives under `azd-ai-agent/infra/` — don't merge with root Bicep files

## Conventions

- Bicep templates use parameter files per environment under `environments/`
- PowerShell deploy scripts are idempotent — safe to re-run
- Helm chart values follow `squad` naming conventions; see `helm/squad/values.yaml` for structure
- Kubernetes manifests use `apps/squad` namespace by convention
- `README-AKS-AUTOMATIC.md` is the canonical runbook for cluster ops
