## What This Area Does

The API area contains the **FedRamp Dashboard ASP.NET Core Web API** -- a .NET backend that
exposes compliance, controls, environments, history, and reports data for the FedRamp Dashboard.

Key responsibilities:
- **Authentication & authorization** via Azure AD / Entra ID (MSAL / `Microsoft.Identity.Web`)
  with RBAC role policies (`SecurityAdmin`, `SecurityEngineer`, `SRE`, `OpsViewer`)
- **PoP Token validation** (MISE V2) -- `PopTokenValidationMiddleware` enforces Proof-of-Possession
  tokens on all authenticated endpoints
- **Data access** via Azure Cosmos DB (`CosmosDbService`) and Azure Log Analytics
  (`LogAnalyticsService`)
- **Cache telemetry** -- `CacheTelemetryMiddleware` + `CacheTelemetryService` track cache hit/miss
  metrics
- **OpenAPI spec** -- `openapi-fedramp-dashboard.yaml` at the area root

This API is the primary data backend for the FedRamp compliance dashboard UI.

## Routing Hints

- **Auth / PoP token changes** -> Data implements; Worf mandatory review
- **RBAC policy changes** (`RbacRoles.cs`, auth policies in `Program.cs`) -> Data + Worf
- **New controllers or endpoints** -> Data implements; update `openapi-fedramp-dashboard.yaml`
- **CosmosDB schema or query changes** -> Data; check for index/RU impact
- **`appsettings.json` changes** -> Data; never commit real secrets -- use managed identity / Key Vault refs
- **Infrastructure wiring** (new services, container config) -> B'Elanna for deployment review
- **OpenAPI contract changes** -> Data authors; Picard signs off on breaking changes

## Constraints

- Never commit real secrets, connection strings, or tokens -- use Azure Key Vault or managed identity
- All new endpoints require OpenAPI spec update (`openapi-fedramp-dashboard.yaml`)
- PoP token validation must not be bypassed -- Worf reviews any auth middleware changes
- Role constants in `RbacRoles.cs` are the source of truth -- do not duplicate inline
- Cosmos DB queries must be reviewed for RU cost on high-frequency paths

## Conventions

- Controllers are thin -- business logic lives in Services
- DI registration is in `Program.cs`; keep it grouped by concern
- Use `ILogger<T>` throughout; no `Console.Write`
- `appsettings.json` contains only non-secret config; secrets via environment / Key Vault
- Integration tests live in `api/FedRampDashboard.Tests/`
